# Plan: Docker Container Smoke Test for ha-dashboard-builder

## Goal

Clone the latest `calzonswe/ha-dashboard-builder` repo, build and start it via Docker Compose, verify the container spins up correctly, and if errors occur, create GitHub issues to track each problem.

---

## Current Context / Assumptions

- **Repo**: `https://github.com/calzonswe/ha-dashboard-builder`
- **Local path**: `/home/calzon/projects/ha-dashboard-builder-test` (fresh clone)
- **Docker & Docker Compose**: Installed and available on this machine.
- **Home Assistant instance**: May or may not be reachable from this machine — the app requires `HA_TOKEN` env var to connect, but the container should still start even without a valid HA connection (it will log errors but won't crash).
- **Dockerfile in use**: The root `Dockerfile` (multi-stage with `uv`). It copies `pyproject.toml`, `uv.lock`, and `app/`.
- **docker-compose.yml**: Uses `build: .` → builds the root Dockerfile. Runs on `network_mode: host`. Exposes port 8000.

---

## Known Issues (from repo inspection)

1. **Missing `uv.lock` file** — The Dockerfile runs `uv sync --no-dev --frozen`, which requires a lock file. If it doesn't exist, the build will fail.
2. **Dockerfile references root-level `app/`** — But the actual FastAPI code lives in `backend/app/`. The root `Dockerfile` does `COPY app/ ./app/`, which would copy nothing (or stale data) since there's no `app/` at repo root.
3. **docker-compose.yml uses `network_mode: host`** — This works on Linux but means the container shares the host network stack; port mapping is redundant but harmless.

## Step-by-Step Plan

### Phase 1: Clone & Inspect

1. Create a fresh test directory: `/home/calzon/projects/ha-dashboard-builder-test`
2. Clone `https://github.com/calzonswe/ha-dashboard-builder.git` into it
3. Verify the clone succeeded and note the commit hash

### Phase 2: Attempt Docker Build & Run

4. Navigate to the cloned repo directory
5. Create a minimal `.env` file with placeholder values (so docker-compose doesn't fail on missing vars):
   ```
   HA_TOKEN=test-token
   HA_INTEGRATION_API_KEY=test-api-key
   ```
6. Run `docker compose up --build 2>&1 | tee /tmp/docker-test.log`
7. Watch for: build errors, container start failures, health check results

### Phase 3: Verify & Diagnose

8. If the build succeeds and container starts:
   - Check logs: `docker logs ha-dashboard-api`
   - Hit the health endpoint: `curl http://localhost:8000/api/health`
   - Record success or any runtime errors

9. If the build fails, capture the exact error output and diagnose root cause(s)

### Phase 4: Create GitHub Issues (if needed)

10. For each distinct problem found, create a GitHub issue via `gh`:
    ```
    gh issue create --title "..." --body "..."
    ```
11. Label issues appropriately (`bug`, `docker`, `build`)
12. Report back with: what worked, what didn't, and the created issue numbers
