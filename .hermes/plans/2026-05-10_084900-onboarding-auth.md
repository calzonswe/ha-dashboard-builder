# Onboarding Wizard & Application Authentication

## Goal

Replace `.env`-based configuration with a GUI-driven first-run onboarding flow, and implement basic application authentication to secure network access.

---

## Current Context

### Existing Architecture
- **Backend**: FastAPI + Pydantic settings loaded from `.env` / environment variables
- **Frontend**: React + TypeScript + Tailwind CSS (Vite dev server)
- **Database**: SQLite (`ha_dashboard.db`) — already in use for dashboard/card persistence
- **Current Config Loading** (`config.py`): `Settings(BaseSettings)` reads HA host/port/token, LLM provider/URL/model from `.env` or env vars
- **Current Connection Flow**: `ConnectionForm.tsx` collects HA host/port/token and sends to `/api/ha/connect`, but config is still read from `.env` at startup — runtime connection doesn't persist settings
- **Settings UI**: `SettingsModal.tsx` exists in frontend for display-only settings view

### What Needs to Change
1. Configuration must be stored in the database, not `.env` files
2. First-run users should see an onboarding wizard instead of the dashboard
3. Application routes require authentication when accessed over the network
4. Runtime HA connection (via `/api/ha/connect`) should persist settings to DB

---

## Proposed Approach

### Phase 1: Persistent Configuration Layer

**Backend changes:**
- Create a `settings` database table/model with fields for all current config values:
  - `ha_host`, `ha_port`, `ha_ssl`, `ha_access_token` (encrypted)
  - `llm_provider`, `llm_base_url`, `llm_model`
  - `app_name`, `created_at`, `updated_at`
- Add CRUD endpoints for settings:
  - `GET /api/settings` — retrieve current config (mask tokens)
  - `PUT /api/settings` — update configuration
  - `POST /api/settings/initialize` — seed initial config during onboarding
- Modify existing routes (`ha/connect`, entity discovery, dashboard creation) to read from DB instead of `.env`
- Keep `.env` as fallback for non-sensitive runtime values (e.g., `DATABASE_URL`)

**Frontend changes:**
- Replace `ConnectionForm.tsx` with a multi-step onboarding wizard component
- Update `SettingsModal.tsx` to allow editing persisted settings
- Add auth state management (JWT/session token)

### Phase 2: Application Authentication

**Backend:**
- Implement JWT-based authentication using `python-jose` or `PyJWT`
- Auth endpoints:
  - `POST /api/auth/login` — accept username/password, return JWT
  - `POST /api/auth/register` — allow first-time admin user creation (during onboarding)
  - `GET /api/auth/me` — current user info
- Add authentication middleware to protect all API routes except:
  - `/api/auth/*` endpoints
  - `/api/health` (if exists)
  - OpenAPI docs (`/docs`, `/openapi.json`) — optionally protected too

**Frontend:**
- Login page component shown when not authenticated
- Auth context/provider for managing session state
- Axios interceptor to attach JWT token to all API requests
- Redirect unauthenticated users to login page

### Phase 3: Onboarding Wizard UX

**Multi-step flow:**
1. **Welcome screen** — app intro, "Get Started" button
2. **Create admin account** — username/password setup (first-run only)
3. **Home Assistant connection** — host, port, token input + test connection button
4. **LLM configuration** — provider selection (Ollama/LMStudio/None), base URL, model
5. **Confirmation** — summary of settings, "Save & Continue" to dashboard

**State management:**
- Track onboarding completion in DB (`settings.onboarded = true`)
- Skip wizard if `onboarded` is already set (show login instead)
- Allow re-accessing setup via Settings modal for configuration changes

## Risks & Tradeoffs

### Security Considerations
- **HA token storage**: Must be encrypted at rest in DB (use `cryptography` library)
- **JWT secret**: Should be generated on first run, not hardcoded
- **Password hashing**: Use `passlib` with bcrypt for user passwords
- **CORS**: Update CORS settings to allow frontend origin after auth is enabled

### Tradeoffs
- **Complexity vs simplicity**: JWT adds auth overhead but is standard and scalable
- **DB vs session storage**: DB persistence survives restarts; in-memory is simpler but ephemeral
- **Onboarding wizard vs inline settings**: Wizard is cleaner for first-run but requires more UI work

### Open Questions
1. Should the onboarding be accessible after initial setup (via Settings), or only shown once?
2. Do we want role-based access (admin vs viewer) or just single-user auth?
3. Should LLM configuration allow multiple providers simultaneously?
4. Is there a preferred auth style — JWT tokens, session cookies, or API key based?
